## 今天最值得关注的 1-3 个产品

### 1. PMB

* 一句话定位：面向 Claude Code、Cursor、Codex 等 coding agent 的本地优先持久记忆层，通过 MCP 暴露给不同 Agent 使用。
* 解决的真实问题：Coding agent 每次换会话、换工具或重启后，都要重新解释项目结构、历史决策、个人偏好和踩坑经验；这会让复杂工程任务停留在“单次上下文”里，难以形成可复用能力。
* 核心机制：官方 GitHub 与 PyPI 信息显示，PMB 把 decisions、lessons、facts 存在本地 SQLite 文件中，通过 MCP 连接 Claude Code、Cursor、Codex、Windsurf、VS Code、Zed 等工具；它强调离线、无 API key、无云端依赖，并提供 recall、prepare、record 等记忆工具。官方 README 提到 29 个 MCP tools、LanceDB 向量存储和 hooks 注入记忆等机制，具体效果指标为官方自述，未做第三方验证。
* 为什么值得关注：它把 Agent memory 从“聊天摘要”拉回到开发者可拥有、可迁移、可审计的本地资产。对 coding agent 来说，记忆不是装饰功能，而是让跨会话任务、项目偏好和团队经验不被反复丢失的基础设施。
* 如果这个产品失败，最可能输在哪里：风险在记忆质量和工作流集成。如果召回结果不够准、写入策略污染上下文，或不同 IDE/Agent 的 MCP 行为不稳定，用户会把它当作额外维护成本，而不是默认底座。
* 对混元 API / Agent 产品的启发：Agent 平台应把 memory 分成事实、决策、偏好、项目结构、任务证据等类型，并提供本地/私有部署、可编辑记忆、召回审计和跨 Agent 迁移，而不是只提供黑盒长上下文。
* 评分：17/18（相关度 5，机制新颖度 5，产品设计启发 5，市场信号 2）
* 官方链接：https://github.com/oleksiijko/pmb ，https://pypi.org/project/pmb-ai/ ，https://www.producthunt.com/products/pmb-local-first-memory-for-ai

### 2. Agent Mode by Receiptor AI

* 一句话定位：面向收据、发票和费用资料的自动化 bookkeeping 工作流，Product Hunt 标题强调 Agent Mode。
* 解决的真实问题：财务资料处理的瓶颈不是单张票据 OCR，而是从邮箱、WhatsApp、上传文件中持续收集材料，抽取字段，按业务语境分类，再同步到 QuickBooks、Xero 或导出成审计可用报告。
* 核心机制：Receiptor 官网与 llms.txt 显示，它会连接 Gmail、Outlook、IMAP、WhatsApp 和手动上传，自动识别收据/发票，抽取 vendor、date、amount、tax、line items、payment method 等字段，并进行分类、规则处理和会计系统同步。Product Hunt 所称 Agent Mode 的具体交互、审批和异常处理边界，本次未能在官网中完整核实。
* 为什么值得关注：它代表的是垂直 Agent 的务实路线：先把高频、规则明确、容错空间有限的文档流跑通，再谈自治。相比泛助手，财务 Agent 更容易定义输入源、字段标准、系统去向和人工复核点。
* 如果这个产品失败，最可能输在哪里：风险在信任、异常处理和系统集成。票据分类、税率、报销规则一旦出错，用户需要回到人工复核；QuickBooks/Xero 客户端配置、历史邮箱扫描权限和多语言/多币种边界也会直接影响使用频率。
* 对混元 API / Agent 产品的启发：垂直 Agent 应把“收集-抽取-分类-同步-复核”做成可追踪 pipeline，并对每一步输出置信度、证据来源和人工确认入口；财务类工具调用尤其需要审计日志和回滚设计。
* 评分：14/18（相关度 4，机制新颖度 3，产品设计启发 4，市场信号 3）
* 官方链接：https://receiptor.ai/ ，https://receiptor.ai/llms.txt ，https://www.producthunt.com/products/receiptor-ai

## 这些产品背后的趋势信号

1. Agent memory 正在从云端黑盒能力转向用户可拥有的本地资产：PMB 是最明确样本，ClinePass 也从另一侧说明 coding agent 生态正在围绕成本、模型和工具链做长期使用优化。
2. 垂直自动化比泛 Agent 更容易落地：Receiptor 和 Sami 都不是让 AI 自由发挥，而是把收集、判断、执行、通知、人工批准限制在明确业务流程里。
3. 今日未出现新的统一协议或大平台范式变化，更多是围绕已有 Agent 工作流补齐记忆、成本、审批和系统连接。

## 我作为 AI 产品经理最想跟进的方向

最想跟进“可拥有、可迁移、可审计的 Agent memory”。现在值得跟进，是因为长任务 Agent 的失败常来自上下文断裂，而不是模型不会推理。建议下一步动作：深入拆解 PMB 的记忆类型、MCP 工具边界、召回策略和本地存储结构，并加入竞品库，和 Zaro、Atlas、note.md、Polygraph 共同比较。
